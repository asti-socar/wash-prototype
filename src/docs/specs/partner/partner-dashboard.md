# 파트너 대시보드

## 1. 개요
- **화면명**: 파트너 대시보드
- **목적**: 자사 배정 오더의 상태별 현황 모니터링, 리스크 오더 사전 감지, 시계열 추이 확인
- **데이터 소스**: `orders.json` + `orders-leadTime.json` (useMemo 실시간 산출)
- **데이터 범위**: `partner === currentPartner.partnerName` 으로 사전 필터링 (자사 데이터만)
- **데이터 갱신**: 수동 (새로고침 버튼), 조회 기준일 오늘 고정

### 인터널 대시보드 대비 차이점
| 항목 | 인터널 | 파트너 |
| :--- | :--- | :--- |
| 타이틀 | "세차 운영 대시보드" | "{partnerName} 대시보드" |
| 서브타이틀 | "실시간 현황 모니터링 및 리스크 관리" | "운영 현황 모니터링 및 리스크 관리" |
| 데이터 범위 | 전체 오더 | 자사 배정 오더만 |
| 파트너 수행 관리 영역 | 있음 (파트너별 테이블) | **제거** (단일 파트너이므로 불필요) |
| 시계열 차트 제목 | "일자별 정규 오더 생성량" | "일자별 오더 생성량" (전체 오더 기준) |
| 네비게이션 대상 | 인터널 오더 관리 (quickFilter) | 파트너 오더 조회 (initialFilter) |
| 영역 수 | 4개 | 3개 (파트너 수행 관리 제거) |

## 2. 화면 구성 및 기능

### 2.1 헤더
| 요소 | 설명 |
| :--- | :--- |
| 타이틀 | "{currentPartner.partnerName} 대시보드" |
| 서브타이틀 | "운영 현황 모니터링 및 리스크 관리" |
| 갱신 시간 | "마지막 갱신: N초 전" 또는 "N분 전" |
| 새로고침 버튼 | RefreshCw 아이콘 + "새로고침", 클릭 시 전체 데이터 갱신 |

### 2.2 [영역 1] 오더 현황

자사 오더의 상태별 분포를 **도넛 차트 + 4개 상태 카드**로 시각화합니다.

#### 도넛 차트
- 4개 상태: 수행 대기(발행+예약), 수행 중, 완료, 취소
- 차트 중앙: 전체 오더 수 + "전체" 라벨
- 범례: 차트 우측 세로 배치 (`● 상태명: 건수`), 클릭 시 오더 조회로 이동

#### 상태 카드 (4개)
| 카드 | 메인 수치 | 하위 항목 | 색상 |
| :--- | :--- | :--- | :--- |
| 수행 대기 | 발행 + 예약 합계 | 발행: N건, 예약: N건 | #7BA3C9 |
| 수행 중 | 수행 중 건수 | - | #E8C47C |
| 완료 | 완료 건수 | 적시수행: N건 (`!isDelayed`), 지연수행: N건 (`isDelayed`) | #7BC9A8 |
| 취소 | 취소 건수 | 취소 유형별 건수 (동적 생성, 고정 순서 표시) | #D98E8E |

- 완료 적시/지연: `orders-leadTime.json`의 `isDelayed` 기준 판정
- 취소 유형: `orders.json`의 `cancelType`에서 동적 집계 (0건 유형 미표시)
- 각 카드/하위 항목 클릭 시 해당 필터로 **파트너 오더 조회** 페이지 이동

#### 취소 유형 표시 순서

고정 순서로 표시하며, 0건인 유형은 숨김 처리합니다.

| # | 취소 유형 | 귀책 하이라이트 |
| :---: | :--- | :---: |
| 1 | 시스템(노쇼 취소) | 빨강 (#D98E8E) |
| 2 | 시스템(미예약 취소) | 빨강 (#D98E8E) |
| 3 | 수행원(개인 사유) | - |
| 4 | 시스템(변경 취소) | - |
| 5 | 시스템(우천 취소) | - |
| 6 | 수행원(차량 없음) | - |
| 7 | 수행원(주차장 문제) | - |
| 8 | 시스템(예약 불가) | - |
| 9 | 수행원(기타) | - |

### 2.3 [영역 2] 리스크 관리

리스크 유형별 오더 현황을 **탭 + 도넛 차트 + 4개 상태 카드**로 모니터링합니다.

#### 탭 구성
| 탭명 | orderType 필터값 | 배지 |
| :--- | :--- | :--- |
| 위생장애 | "위생장애" | 건수 |
| 고객피드백(ML)_긴급 | "고객 피드백(ML)_긴급" | 건수 |
| 초장기미세차 | "초장기 미세차" | 건수 |

- 선택 탭: 파란 텍스트(#0052CC) + 언더라인 + 파란 배지
- 미선택 탭: 회색 텍스트(#6B778C) + 회색 배지

#### 리스크 도넛 차트
- 리스크 3종의 전체 건수 표시
- 차트 중앙: 리스크 총합 + "리스크" 라벨

#### 탭별 상태 카드
- 오더 현황과 동일한 4개 카드 구조 (수행 대기, 수행 중, 완료, 취소)
- 취소 카드: 0건인 유형은 숨김
- 각 항목 클릭 시 `{ status, orderType }` 조합으로 오더 조회 이동

### 2.4 [영역 3] 시계열 트렌드 그래프

2열 레이아웃으로 최근 7일 추이를 시각화합니다.

#### A. 일자별 오더 생성량 (막대 그래프)
- X축: 날짜 (MM/DD), Y축: 건수
- **자사 전체 오더** 대상 (인터널은 "정규 오더"만 집계하지만, 파트너는 전체 오더 집계)
- 막대 색상: rgba(123, 163, 201, 0.8)
- 호버 시 툴팁: 날짜, 오더 N건

#### B. 일별 리스크 유형별 추이 (라인 차트)
| 라인 | 색상 |
| :--- | :--- |
| 위생장애 | rgba(217, 142, 142, 0.8) |
| ML긴급 | rgba(232, 196, 124, 0.8) |
| 초장기미세차 | rgba(123, 163, 201, 0.8) |

- 범례: 차트 상단 우측, 원형 아이콘
- 호버 시 툴팁: 날짜, 유형별 건수

## 3. 데이터 산출 로직

모든 통계는 `orders.json` + `orders-leadTime.json`에서 `useMemo`로 실시간 산출합니다. 데이터는 `partner === currentPartner.partnerName` 조건으로 사전 필터링합니다.

### 3.1 `aggregateStatus(orders)` — 공통 집계 함수
| 필드 | 산출 | 비고 |
| :--- | :--- | :--- |
| total | `orders.length` | |
| issued | `status === '발행'` 건수 | |
| reserved | `status === '예약'` 건수 | |
| in_progress | `status === '수행 중'` 건수 | |
| completed.total | `status === '완료'` 건수 | |
| completed.on_time | 완료 중 `!leadTimeData[orderId].isDelayed` | |
| completed.delayed | 완료 중 `leadTimeData[orderId].isDelayed` | |
| cancelled.total | `status === '취소'` 건수 | |
| cancelled.byType | `cancelType`별 건수 맵 | 동적 키 |

- **오더 현황**: `aggregateStatus(partnerOrders)` (자사 전체)
- **리스크 관리**: `aggregateStatus(partnerOrders.filter(o => o.orderType === 유형))` (유형별 3종)

### 3.2 일자별 차트 데이터
- **오더 생성량**: `createdAt.slice(0,10)` 기준 최근 7일 일자별 count (자사 전체 오더)
- **리스크 추이**: 리스크 3종(`위생장애`, `고객 피드백(ML)_긴급`, `초장기 미세차`)의 `createdAt` 기준 최근 7일 일자별 count

### 3.3 네비게이션
- 카드/범례/하위 항목 클릭 시 `goOrdersWithFilter(filter)` 호출
- PartnerApp.jsx에서 `orderFilter` 상태 설정 후 `partner-orders` 페이지로 전환
- 오더 조회 페이지에서 `initialFilter` prop으로 수신
